# Framework source labels

Daily Rhythm Companion exposes small app-facing source labels so the demo can explain where advice came from without leaking implementation details, provider payloads, prompts, keys, tokens, or private local paths.

This document is part of v1.3.0 Framework / LLM configured demo hardening.

---

## Source label contract

The `/advice` response may include `source` metadata.

Current `AdviceSource.engine` values are:

```text
mock
framework
framework_fallback
```

These labels are intentionally small and app-facing.

They should answer one question:

```text
What generated the advice text that the user is seeing?
```

They should not expose:

```text
provider API keys
raw prompts
raw provider payloads
authorization headers
token values
private local paths
```

---

## Label meanings

### `mock`

Use `AdviceSource.engine=mock` when the advice text was generated by DRC's deterministic local mock engine.

Expected meaning:

```text
The response is safe local demo output.
No AI Character Framework checkout was required.
No external LLM provider was called.
```

Saved DailyRecord suffix:

```text
+mock
```

Example basis:

```text
sleep+mood+character+mock
```

### `framework`

Use `AdviceSource.engine=framework` when the advice text was generated through AI Character Framework and a framework text session returned the displayed response.

Expected meaning:

```text
DRC routed the request through FrameworkConversationEngine.
AI Character Framework created or reused a text session.
The displayed text came from the framework session response.
```

Saved DailyRecord suffix:

```text
+framework
```

Example basis:

```text
sleep+mood+character+framework
```

Framework responses may also include:

```text
framework_preset
framework_character
framework_character_source
```

These fields are debug metadata for configured local/demo verification. They must stay free of secrets and private local paths.

### `framework_fallback`

Use `AdviceSource.engine=framework_fallback` when framework mode was selected but DRC could not get a usable framework response, so it returned mock-engine advice through the framework fallback path.

Expected meaning:

```text
The operator selected framework mode.
The framework path was unavailable, failed, or returned unusable output.
The user-visible advice text came from the local mock fallback.
```

Saved DailyRecord suffix:

```text
+framework_fallback
```

Example basis:

```text
sleep+mood+character+framework_fallback
```

Fallback UI wording should be honest. It may say that framework mode fell back to local demo advice. It must not say that configured LLM generation succeeded.

## Configured LLM skip is not an advice source

`configured_llm_skip` is not an `AdviceSource.engine` value.

configured_llm_skip is not an `AdviceSource.engine` value.

Configured LLM skip is an operator-check state printed by configured-only smoke checks when a prepared FW/LLM environment is missing explicit opt-in gates or provider credentials.

Examples:

```text
[v130-framework-llm-configured-smoke] SKIP: FRAMEWORK_ROOT/FRAMEWORK_PROJECT_ROOT is not configured.
[v130-framework-llm-configured-smoke] SKIP: Configured FW/LLM ask smoke requires explicit opt-in.
```

A skip state means:

```text
The smoke check did not run the configured FW/LLM ask path.
No external provider call should have happened.
No user-facing advice was generated by that skipped check.
```

Do not store configured smoke skip states as DailyRecord advice sources.

## User-facing wording guidance

Use source-aware wording in operator docs and debug UI.

Recommended wording:

```text
mock: Local mock advice was used.
framework: Framework-backed advice was used.
framework_fallback: Framework mode was selected, but local fallback advice was used.
configured smoke skip: The configured LLM smoke was skipped because required opt-in setup is missing.
```

Avoid wording like:

```text
LLM advice succeeded
Framework generated this response
Provider-backed response was returned
Real configured LLM mode is active
```

unless the source is actually `framework` and the configured environment intentionally exercised that path.

## Verification commands

Mock-safe source-tree verification:

```powershell
python scripts\check_v130_framework_llm_configured_demo_day4.py
```

Recommended Day1-Day4 verification:

```powershell
python -m compileall -q backend scripts
python scripts\check_v130_framework_llm_configured_demo_day1.py
python scripts\check_v130_framework_llm_configured_demo_day2.py
python scripts\check_v130_framework_llm_configured_demo_day3.py
python scripts\check_v130_framework_llm_configured_demo_day4.py
```

The Day4 check uses local fake framework/runtime fixtures. It does not require a real framework checkout, provider credentials, or external LLM calls.

## Operator checklist usage

The [Framework-backed advice operator checklist](framework_advice_operator_checklist.md) uses this source-label contract when reviewing `/advice`, DailyRecord save, and History output.

The operator should record only safe summaries such as:

```text
/advice source label: framework_fallback
DailyRecord basis suffix: +framework_fallback
History review: OK
optional configured ask smoke: SKIP
```

The operator should not paste provider keys, raw provider payloads, authorization headers, local token files, private absolute paths, or full raw LLM debug traces into shared notes.
