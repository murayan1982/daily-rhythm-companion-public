## Purpose

```text
- Connect the accepted T-1b controller to HomeScreen.
- Replace external audio URL launch with app-owned play, stop, and replay controls.
- Add one concrete cross-platform Flutter audio engine behind the accepted interface.
- Convert missing/gone Backend audio responses into the explicit expired state.
- Keep opaque DRC URLs, private paths, provider payloads, and raw exception text out of normal UI.
- Preserve the accepted 86400-second TTL, 100-artifact cap, MP3-only handoff,
  no-store, and nosniff Backend boundaries without Backend runtime changes.
```

## Concrete audio boundary

T-1c adds:

```text
app/lib/services/audioplayers_voice_output_audio_engine.dart
```

The implementation uses `audioplayers: ^6.7.1`. This is the newest project-compatible
line selected for the accepted Flutter 3.41.7 toolchain; `6.8.x` requires a newer
Flutter SDK. It keeps the required Web/Windows/mobile audio APIs while plugin-specific
objects remain inside the engine/driver implementation and are not exposed to
HomeScreen or the T-1b controller.

The engine downloads the DRC-owned opaque URL through the existing `http` client,
accepts HTTP 200/206 MP3 responses, and gives the resulting bytes to the platform
player. It maps:

```text
HTTP 404 / 410              -> audio_artifact_not_found / expired
other non-success HTTP      -> audio_http_<status> / failed
network request failure     -> audio_http_request_failed / failed
non-MP3 content type        -> unsupported_audio_content_type / failed
empty artifact              -> empty_audio_artifact / failed
platform 404/410/not-found  -> audio_artifact_not_found / expired
other platform failure      -> audio_platform_playback_failed / failed
```

The normal user message never contains the opaque URL, original filename,
Framework artifact path, provider response, or platform exception text.

## HomeScreen integration

`HomeScreen` owns one `VoiceOutputAudioPlayerController` for its lifetime. The
controller uses an injected engine in tests and the concrete audioplayers engine
in the normal app.

The generated URL handoff now shows:

```text
アプリ内音声プレイヤー
音声を再生する
停止する
もう一度再生する
音声を作り直す
```

Behavior:

```text
idle       show play
loading    show stop
playing    show stop
stopped    show replay
completed  show replay
failed     show replay when the source remains available
expired    clear replay source and show regenerate
```

Submitting a new voice-output request resets/stops the previous player state.
Disposing HomeScreen removes the listener and disposes the controller/engine.
Technical codes appear only under `開発者向け再生詳細`; raw URLs stay hidden.

## Mock-safe tests

Focused engine tests:

```text
app/test/audioplayers_voice_output_audio_engine_test.dart
```

They inject an HTTP mock and fake platform driver to cover successful MP3 loading,
play/stop/seek, 404, 410, retryable HTTP failure, content-type rejection, event
forwarding, and platform not-found mapping. They do not initialize an audio device.

Focused widget tests:

```text
app/test/voice_output_audio_player_widget_test.dart
```

They inject a fake BackendApiClient and fake audio engine to cover play/stop/replay,
completion, expiry, direct regeneration, relative opaque URL resolution, and raw
URL non-display. Existing voice-output widget regressions are updated for the
in-app controls.

## Windows scaffold tracking follow-up

The local Windows build also requires the Flutter-managed wrapper scaffold file
`app/windows/flutter/CMakeLists.txt`. The repository-root Flutter SDK safety
ignore is therefore scoped as `/flutter/` rather than the unscoped `flutter/`,
so the app-owned Windows platform directory remains trackable while a root-level
Flutter SDK checkout is still excluded.

This is a source/scaffold correction only. `app/windows/flutter/ephemeral/`
remains ignored and build outputs remain uncommitted.
