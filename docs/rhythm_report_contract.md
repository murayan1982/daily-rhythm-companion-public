# Rhythm report contract

## Position

This document defines the v1.6.0 weekly/monthly rhythm report contract before adding new backend or Flutter runtime implementation.

Day3 chooses a generic `RhythmReport` contract with a `period` field instead of creating unrelated weekly and monthly report contracts.

The existing `WeeklySleepSummary` endpoint remains valid. A later implementation may either keep it as the current weekly card surface or adapt its values into the broader `RhythmReport` shape.

## Contract decision

Use one report contract:

```text
RhythmReport
```

Required period values:

```text
weekly
monthly
```

Do not introduce separate user-facing contracts such as `WeeklyRhythmReport` and `MonthlyRhythmReport` unless a later implementation finds a concrete reason. The first implementation should keep the shape shared so Flutter can render weekly and monthly cards with the same UI pattern.

## Proposed backend model fields

The future backend model should be small, deterministic, and history-derived.

Recommended fields:

```text
period: str
reference_date: str
range_start: str
range_end: str
label: str
total_record_count: int
usable_sleep_record_count: int
average_total_sleep_minutes: int | None
record_dates: list[str]
display_title: str
display_label: str
display_summary: str
display_coverage: str
display_note: str
action_hint: str
source_label: str
data_scope: str
data_quality: str
is_medical_advice: bool
```

Field meaning:

```text
period:
  weekly or monthly.

reference_date:
  The date used as the report anchor.

range_start / range_end:
  Inclusive saved-history date window covered by the report.

label:
  A stable machine-readable summary label for deterministic tests.

total_record_count:
  Number of saved DailyRecord entries in the window, whether sleep is available or unavailable.

usable_sleep_record_count:
  Number of records with available sleep and positive total_sleep_minutes.

average_total_sleep_minutes:
  Average from usable sleep records only. Use null/None when there is not enough usable data.

record_dates:
  Saved DailyRecord dates included in the report window, not raw provider event timestamps.

display_title / display_label / display_summary / display_coverage / display_note / action_hint:
  Presentation-ready conservative copy generated by the service.

source_label:
  Human-readable source label for UI/debug presentation.

data_scope:
  Stable machine-readable scope label.

data_quality:
  Stable machine-readable quality label.

is_medical_advice:
  Always false for v1.6.0 rhythm reports.
```

## Stable labels

Recommended `label` values:

```text
insufficient_data
weekly_short
weekly_balanced
weekly_enough
monthly_sparse
monthly_mixed
monthly_stable
monthly_enough
```

The label is not a medical score. It is a deterministic presentation hint based on saved history coverage and sleep-duration averages.

Recommended `data_quality` values:

```text
insufficient
partial
usable
```

Recommended `data_scope` values:

```text
weekly_history
monthly_history
```

## Source-label policy

Rhythm reports should clearly say they are made from saved DailyRecord history.

Recommended `source_label` values:

```text
saved_daily_record_history
saved_daily_record_history_with_mock_sleep
saved_daily_record_history_with_real_sleep
insufficient_saved_history
```

Source-label rules:

```text
- Use saved_daily_record_history when source kind is mixed or not worth exposing in detail.
- Use saved_daily_record_history_with_mock_sleep when usable records are all mock/sample data.
- Use saved_daily_record_history_with_real_sleep when usable records are real-data records already saved as DailyRecords.
- Use insufficient_saved_history when there are too few saved records to summarize.
- Do not imply that the report directly queried Google Health, Fitbit, or another provider unless that behavior is explicitly implemented later and guarded.
- Do not expose raw provider payloads, tokens, authorization headers, local data paths, or private prompts.
```

## Period windows

Recommended first-pass windows:

```text
weekly: 7 days including reference_date
monthly: 30 days including reference_date
```

The first implementation should prefer simple rolling windows over calendar-month semantics. Calendar month reports can be added later if the UI needs them.

Coverage wording should explicitly mention the window and usable record count, for example:

```text
過去7日分の保存済み記録から、使える睡眠記録3件をもとにした軽い振り返りです。
過去30日分の保存済み記録から、使える睡眠記録8件をもとにした軽い振り返りです。
```

## Sparse-history behavior

Reports must handle sparse or unavailable history without error.

Recommended thresholds:

```text
weekly usable threshold: 2 usable sleep records
monthly usable threshold: 5 usable sleep records
```

When the threshold is not met:

```text
label = insufficient_data
data_quality = insufficient
average_total_sleep_minutes = None
is_medical_advice = False
```

Allowed sparse-history copy:

```text
記録が少ないので、今は参考メモとして見てください。
保存済みの睡眠記録がもう少し増えると、週の傾向を出しやすくなります。
保存済みの睡眠記録がもう少し増えると、月のリズムを振り返りやすくなります。
```

Avoid sparse-history copy:

```text
睡眠状態が悪いです。
改善が必要です。
病気の可能性があります。
今日の睡眠は不足しています。
diagnosis-like labels
treatment advice
```

## Unavailable-history behavior

Unavailable sleep summaries can count toward `total_record_count`, but not toward `usable_sleep_record_count` or sleep-duration averages.

Unavailable data should be explained as history coverage, not as a user health state.

Allowed unavailable-history copy:

```text
保存済み記録はありますが、睡眠時間として集計できる記録が少なめです。
今日の睡眠としては扱わず、過去の保存済み記録からの参考表示です。
```

## Weekly compatibility

The existing weekly surface remains a valid foundation:

```text
WeeklySleepSummary
GET /daily-records/weekly-summary
WeeklySleepSummaryService
```

Day4 can either:

```text
- add RhythmReport alongside WeeklySleepSummary, or
- add a small adapter that maps WeeklySleepSummary into RhythmReport-compatible copy.
```

Day3 does not rename the existing weekly endpoint.

## Monthly implementation direction

The first monthly report should use the generic `RhythmReport` shape with:

```text
period = monthly
data_scope = monthly_history
```

Suggested endpoint options for Day4/Day5 discussion:

```text
GET /daily-records/rhythm-report?period=weekly
GET /daily-records/rhythm-report?period=monthly
```

A separate endpoint such as `GET /daily-records/monthly-summary` is still acceptable later, but the app-facing model should remain compatible with the generic contract unless there is a strong reason to split it.

## Non-goals

Day3 does not add runtime implementation.

Out of scope for Day3:

```text
- backend/app/models/rhythm_report.py implementation
- rhythm report service implementation
- API route implementation
- Dart model implementation
- Flutter History UI card implementation
- DailyRecord schema changes
- real provider API calls
- external LLM calls
- AI Character Framework requirement
- release zip creation or rebuild
```

## Day3 conclusion

v1.6.0 should proceed with a generic `RhythmReport` contract using `period=weekly/monthly`, saved DailyRecord history, explicit source labels, stable data-quality labels, and conservative non-medical wording.
