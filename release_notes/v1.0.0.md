# Daily Rhythm Companion v1.0.0

Daily Rhythm Companion v1.0.0 is the first release candidate focused on making the project usable as a **smartphone-web-testable AI Character Framework demo app**.

This release is not a store release or a production hosted consumer service. It is a public demo-app readiness milestone: the app, docs, scripts, and release package are organized so a developer or demo operator can understand the project, run it safely in mock mode, and verify the v1.0 demo flow.

---

## Highlights

- Positioned Daily Rhythm Companion as a full demo app for **AI Character Framework** integration.
- Defined the v1.0 user-facing direction: **try the app from a smartphone browser**.
- Added public repository readiness policies for docs, scripts, and release package hygiene.
- Added smartphone web demo verification guidance.
- Added Google Health real API **safe default + explicit opt-in** policy.
- Added v1.0 release package and final release checks.
- Added compatibility/final sweep checks for fixed release zip verification.
- Preserved mock-safe local development as the default path.

## What changed

### Smartphone Web demo readiness

v1.0 documents the expected smartphone web demo flow:

```text
Open the DRC Web UI from a smartphone browser
→ connect to the backend API
→ view sleep/context
→ select mood
→ select character
→ generate advice
→ save the result as a DailyRecord
→ review it in History
→ understand capability status and fallback states
```

The docs clarify that a phone cannot use the development PC's `127.0.0.1`. A smartphone demo needs a LAN IP address, a tunnel URL, or a deployed HTTPS URL.

### Public repository readiness

The repository now includes public-facing docs for:

```text
README.md
app/README.md
docs/quickstart_local.md
docs/quickstart_smartphone_web.md
docs/framework_demo_setup.md
docs/google_health_real_api_opt_in.md
docs/troubleshooting.md
scripts/README.md
```

The v1.0 direction is:

```text
Daily Rhythm Companion
= smartphone-web-testable demo app
+ public repository demo for AI Character Framework
```

### Script and cleanup policy

The scripts directory is now documented by purpose:

```text
- everyday mock-safe checks
- v1.0 public readiness checks
- release package and final release checks
- configured-only demo checks
- versioned compatibility checks
```

Cleanup is governed by:

```text
KEEP
MERGE
ARCHIVE
DELETE
```

Old versioned checks are not deleted blindly. They remain compatibility/reference material until v1.0 checks replace their coverage.

### Google Health explicit opt-in

v1.0 keeps the default local profile mock-safe.

The intended policy is:

```text
Safe default + documented explicit opt-in.
Guarded does not mean unimplemented.
```

Real Google Health API calls must not run without explicit opt-in. The real API path must remain separate from normal local development, and token / secret / raw payload / local path exposure is not allowed.

The tracked `backend/env_profiles/mock_safe.env` is treated as a safe public env profile only when real API opt-in flags remain disabled.

### Release package readiness

v1.0 release mode and handoff mode are documented separately.

Release mode is intended to be a public/demo artifact:

```text
- README.md
- build_release.bat
- backend/
- app/
- selected public docs
- selected v1.0 readiness docs
```

Release mode intentionally excludes root `scripts/`, local data, secrets, tokens, cache, virtual environments, and build artifacts.

Handoff mode remains for development handoff and can include docs and scripts.

## Known limitations

v1.0 is a demo/public-repo readiness release. It intentionally does not claim:

```text
- App Store / Google Play distribution
- production hosted consumer-service readiness
- production-grade Google Health onboarding
- mandatory microphone support
- mandatory TTS provider support
- mandatory Live2D/VTS runtime
- always-on realtime assistant behavior
```

Configured features such as AI Character Framework-backed LLM responses, Google Health real API access, voice input, voice output/TTS, and Live2D/VTS motion should remain opt-in and environment-dependent.

The smartphone web demo still requires an access model that a phone can reach, such as LAN IP, tunnel URL, or deployed HTTPS URL.
